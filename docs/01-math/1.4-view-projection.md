## 1. 다시 보는 여정 — 이제 코드로

```
점 ──(Model)──► 월드 ──(View)──► 뷰 ──(Projection)──► 클립 ──(÷w)──► NDC ──► 화면
```

- **Model (1.3에서 완성):** 큐브를 월드의 원하는 곳에 배치.
- **View (이번):** 카메라 기준으로 세상을 옮김.
- **Projection (이번):** 원근감을 주고 화면 상자로 밀어넣음.
- **÷w (이번):** GPU가 자동으로 해주지만, 우리는 그 원리를 이해합니다.

세 행렬을 곱한 게 **MVP**이고, 정점 셰이더가 하는 일이 딱 이 한 줄입니다: `clip = v * MVP`.

---

## 2. 뷰 행렬 — "카메라를 원점으로"

핵심 발상: GPU는 항상 **원점에 있는 가상의 눈**에서 렌더링합니다. 그런데 우리 카메라는 `(0,1,-4)` 같은 곳에 있죠.
그래서 **"카메라가 원점에 오도록 온 세상을 반대로 옮기는"** 변환이 뷰 행렬입니다.

뷰 행렬을 만들려면 카메라의 **좌표축 3개**가 필요합니다. 1.1의 외적이 여기서 빛을 발합니다.

```cpp
Vector3 zaxis = forward.Normalized();           // 앞 (카메라가 보는 방향)
Vector3 xaxis = up.Cross(zaxis).Normalized();   // 오른쪽 = 위 × 앞   (왼손 좌표계!)
Vector3 yaxis = zaxis.Cross(xaxis);             // 진짜 위 = 앞 × 오른쪽
```

> **왜 `up`을 그대로 안 쓰고 다시 계산?** 사용자가 준 `up`은 대략적인 "위"일 뿐, `forward`와 정확히 수직이 아닐 수 있습니다.
> 외적으로 `right`를 구하고, 다시 외적으로 **forward에 정확히 수직인 진짜 up**을 만듭니다. 이렇게 셋이 서로 수직인 깔끔한 좌표축이 됩니다.

이 축들을 행렬에 배치하면(회전 부분) + 카메라 위치를 빼면(이동 부분) 뷰 행렬 완성:

```cpp
Matrix r;
r._11=xaxis.x; r._12=yaxis.x; r._13=zaxis.x;    // 축을 '열'로
r._21=xaxis.y; r._22=yaxis.y; r._23=zaxis.y;
r._31=xaxis.z; r._32=yaxis.z; r._33=zaxis.z;
r._41=-xaxis.Dot(eye); r._42=-yaxis.Dot(eye); r._43=-zaxis.Dot(eye);  // -eye 투영
```

> **순서 주의 (단골 버그):** `up.Cross(zaxis)`와 `zaxis.Cross(up)`은 정반대 방향입니다(1.1의 외적 순서 성질).
> 틀리면 화면이 좌우로 뒤집힙니다. 우리 규약(왼손)에서는 `right = up × forward`.

## 3. 투영 행렬 — 원근감의 핵심

원근 투영은 두 가지 일을 합니다: **(1) 시야각만큼 화면에 맞게 스케일** + **(2) 깊이(z)를 w에 복사**.

```cpp
float yScale = 1.0f / tan(fovY * 0.5f);   // 시야각이 좁을수록 yScale↑ (망원 줌)
float xScale = yScale / aspect;            // 화면 가로:세로 비율 보정

         xScale  0       0                 0
P  =     0       yScale  0                 0
         0       0       zf/(zf-zn)        1     ← _34 = 1 이 핵심!
         0       0       -zn*zf/(zf-zn)    0
```

`_34 = 1`이 마법입니다. `(x,y,z,1) * P`를 계산하면 **결과의 w 성분에 뷰 공간의 z가 들어갑니다.**

```
clip.w = x·0 + y·0 + z·1 + 1·0 = z      (뷰 공간 깊이)
```

데모에서 이걸 눈으로 확인할 수 있어요:

```
가까운 점 z=0 :  w = 4.00      (카메라가 -4에 있으니 뷰 공간 깊이 4)
먼   점 z=8 :  w = 12.00      (뷰 공간 깊이 12)
```

**w가 바로 "카메라로부터의 거리"**가 된 겁니다.

---

## 4. ★ ÷w — 멀수록 작아지는 이유

클립 공간 좌표 `(x, y, z, w)`를 **w로 나누면** 정규화 장치 좌표(NDC, -1~1)가 됩니다. 이걸 **원근 나눗셈**이라 합니다.

```
NDC = (x/w, y/w, z/w)
```

w가 "거리"였으니, **멀리 있는 점일수록 큰 수로 나눠져 화면 중앙(0)에 가까워집니다.** 이게 원근감의 전부입니다.

데모 결과:

```
가까운 점 z=0 :  NDC.x = 0.339   (w=4  로 나눔)
먼   점 z=8 :  NDC.x = 0.113   (w=12 로 나눔)   ← 같은 x=1 인데 화면에선 중앙에 가깝게 = 작게
```

같은 월드 x좌표(=1)를 가진 두 점인데, 먼 점이 화면에서 더 중앙(작게)에 찍힙니다. **철길이 멀리서 모이는** 바로 그 효과죠.

```cpp
Vector3 Vector4::ToVector3DivW() const {
    if (w == 0.0f) return { x, y, z };
    return { x / w, y / w, z / w };
}
```

> **GPU가 대신 해줍니다.** 실제 렌더링에서 정점 셰이더는 `clip`(÷w 전)을 출력하고, GPU 래스터라이저가 ÷w를 자동으로 합니다.
> 우리는 셰이더에서 `mul(v, MVP)`만 쓰면 됩니다(Part 3~4). 하지만 **왜 그게 동작하는지** 이제 알게 됐습니다.

## 5. ★ MVP 조립

이제 전부 합칩니다. 행벡터 규약이라 **왼쪽부터 적용 순서**대로:

```cpp
Matrix mvp = model * view * proj;        // 점을 모델→뷰→투영
Vector4 clip = Vector4{ pos, 1.0f } * mvp;
```

이 `mvp` 행렬이 곧 Part 3~4의 정점 셰이더에 상수 버퍼로 올라가고, HLSL에서 이렇게 쓰입니다:

```hlsl
cbuffer Transform { row_major float4x4 gMVP; };
o.pos = mul(float4(input.pos, 1.0f), gMVP);   // 우리가 만든 v * MVP 그대로!
```

> **규약의 보상:** 우리가 처음부터 "행우선 저장 + 행벡터"로 통일했기에, CPU에서 만든 `mvp`를 **전치 없이 그대로** GPU에 올려
> HLSL `row_major` + `mul(v, M)`로 쓸 수 있습니다. 0.2에서 약속했던 실용적 이점이 바로 이것입니다.

## 6. Part 1 정리

축하합니다. 그래픽스 수학의 **뼈대 전체**를 직접 만들었습니다.

```
벡터(1.1) → 행렬(1.2) → 변환(1.3) → 뷰·투영·MVP(1.4)
  내적/외적   곱/합성     이동·회전     카메라·원근·÷w
```

이 `Math` 라이브러리는 앞으로 **고치지 않고 계속 씁니다.** Part 3에서 삼각형을 띄울 때, Part 4에서 큐브를 회전시킬 때, 여기서 만든 `mvp`가 그대로 GPU로 갑니다.
